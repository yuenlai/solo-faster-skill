# solo-faster v2 单仓批量执行

solo-faster v2 的目标是把一个代码仓库当成唯一执行对象：先按默认分布或自定义分布批量生成主提示词，写入父目录 `<仓库目录名>-solo-faster-prompts.xlsx`，再在同一个 Trae 窗口按主提示词编号升序串行投递、等待用户人工确认已跑完、验收、回填、逐轮提交推送 git，并进入下一个主提示词。

不要真实跑 Trae 的场景必须只做静态检查、脚本自检和 dry-run。dry-run 可以创建临时 workbook、模拟状态变化、模拟超时和断点续跑，但不能粘贴或发送到 Trae。

## 单仓批量执行路

核心对象：

- 目标仓库：一次只处理一个代码仓库。
- 主提示词：批量生成阶段最开始生成的提示词。
- 主提示词编号：从 `1` 开始连续编号。
- 实际执行序号：指“当前执行到的第几个主提示词”，只统计 `行类型=主提示词` 的行，并按当前 workbook 里的实际执行顺序计算；不看修复提示词行，不把整张表行号当成这个序号，也不能拿 `主提示词编号` 直接代替。
- 轮次：主提示词为第 `1` 轮，修复提示词为第 `2` 或第 `3` 轮。
- 行类型：`主提示词` / `修复提示词`。
- Bug 修复行数：同一仓库内任务类型为 `Bug修复` 的 Excel 行数。

仓库选择规则：

- 当前目录是 git 仓库或仓库内部目录时，直接处理该仓库根目录。
- 当前目录是父目录且只有一个代码子目录时，选中该子目录。
- 当前目录下有多个候选仓库时暂停，让用户指定；不要猜。
- workbook 默认命名为 `<仓库目录名>-solo-faster-prompts.xlsx`，放在父目录，仓库恢复不能删除或覆盖它。
- 兼容旧文件名 `solo-faster-prompts.xlsx`；如果父目录里已经有旧文件名，脚本默认仍能读到它。

仓库解析和 workbook 初始化命令：

```bash
python3 $CODEX_HOME/skills/solo-faster/scripts/batch_prompt_workbook.py scan --parent "<父目录>" --repo "<仓库路径>" --difficulty-mode 中
```

真实执行前先打开 Trae 到目标仓库目录：

```bash
open -a "/Applications/Trae CN.app" "<仓库路径>"
```

要求：

- 这一步发生在第一次提示用户发送主提示词之前，不要等到用户开始发第 1 条时才想起来打开。
- 默认把 `open -a "/Applications/Trae CN.app" "<仓库路径>"` 当成单仓真实执行流程的起手动作。
- 如果本机 Trae 实际安装名不是 `Trae CN.app`，先定位真实 app 名称，再用对应名称打开；不要跳过这一步。
- 打开后要确认前台窗口属于 Trae，且当前工作区已经切到目标仓库目录；如果没切对，要先修正，再继续后面的 workbook 断点续跑和发送提示。
- 只有静态检查、dry-run、自检场景才可以不真实打开 Trae。

手动指定主提示词类型与条数时，使用 `--task-counts`：

```bash
python3 $CODEX_HOME/skills/solo-faster/scripts/batch_prompt_workbook.py scan --parent "<父目录>" --repo "<仓库路径>" --difficulty-mode 中 --task-counts "0-1代码生成=20,Feature迭代=20,代码理解=1,代码重构=1,工程化=1"
```

dry-run 可用占位提示词验证默认或自定义分布：

```bash
python3 $CODEX_HOME/skills/solo-faster/scripts/batch_prompt_workbook.py scan --parent "<临时父目录>" --repo "<仓库路径>" --with-stubs --force
```

## 并发约束

`solo-faster` 现在区分两类阶段：

- 静态阶段：允许并行执行，包括仓库识别、scan、主提示词生成、Excel 初始化与查看、dry-run、自检、只读状态查询。
- 真实执行阶段：包括提示用户手动发送、回填 `Trae Session ID`、验收、git commit、git push。

规则：

- 在当前默认人工流程下，不启用锁。
- 前提是：一个项目仓库只对应自己的 workbook，并且同一项目只会单个跑。
- 在这个前提下，不同项目之间互不共享 Excel，也不共享 git 仓库，因此默认不需要额外锁。
- `scan`、`status`、`dry-run`、提示用户发送、等待用户回 `Trae Session ID`、等待用户回复“继续”都按无锁流程处理。
- 只有在显式启用 `--auto-ui`、`--active-monitor` 或 `new-task` 这类 Trae UI 自动化路径时，才允许脚本临时拿运行锁；默认人工模式不要拿锁。
- 如果以后恢复成“同一项目可能被多个线程同时推进”，或重新引入自动化发送/自动化新建任务，再单独设计项目级锁；不要恢复跨项目全局锁。

## Excel 数据模型

父目录维护 `<仓库目录名>-solo-faster-prompts.xlsx`。新版 Excel 必须包含并只按下面列语义执行：

- `仓库路径`
- `主提示词编号`
- `轮次`
- `行类型`
- `执行状态`
- `Repo ID`
- `Trae Session ID`
- `提示词`
- `任务类型`
- `业务领域`
- `修改范围`
- `任务难度`
- `任务是否完成`
- `过程与产物是否满意`
- `不满意原因`
- `Repo URL`
- `Commit ID`
- `基线ID`
- `开始时间`
- `结束时间`
- `更新时间`
- `备注`

Excel 回填硬规则：

- 任何一轮只要发生状态变化、拿到新的确定字段、完成验收、完成 git commit、完成 git push，必须立刻写回 Excel；不能先继续流程，之后再补。
- 回填要求是“当前时点相关字段完整回填”，不是只改一个 `执行状态` 就算完成；已经确定的 `Trae Session ID`、`任务是否完成`、`过程与产物是否满意`、`不满意原因`、`修改范围`、`Repo URL`、`Commit ID`、`开始时间`、`结束时间`、`更新时间`、`备注` 等字段，都必须同步补齐。
- 只要当前轮已经完成 git push，就必须先把该轮真实 `Commit ID` 回填到 Excel，确认落盘成功后，才能继续 `next`、发送下一条提示词、进入下一轮验收或通知用户下一步。
- 任一应回填字段漏填，都视为当前轮未真正收尾；必须先补齐 Excel，再继续后续流程。

字段枚举：

- `Repo ID` 规则固定为：`当前仓库目录名-主提示词编号`。同一主提示词下的修复轮次必须与第 1 轮使用完全相同的 `Repo ID`。
- `Trae Session ID` 默认先留空；当前轮提示词一旦发送成功，就立刻在对应 Trae 任务里找到这条提示词后紧接的 `SOLO Agent` 回复，复制该轮自己的完整值并回填。拿不到 `Trae Session ID` 的这一轮不能继续推进，视为无效执行。
- `任务类型` 只能是：`Bug修复`、`0-1代码生成`、`Feature迭代`、`代码理解`、`代码重构`、`工程化`、`代码测试`。
- `业务领域` 只能是：`全栈 Web 应用`、`Web 前端`、`纯后端服务`、`命令行工具`、`科学计算`、`3D / 交互可视化`、`游戏开发`、`桌面应用（含 GUI）`、`AI/ML 应用`、`数据分析与可视化`、`自动化与工具脚本`。
- `修改范围` 只能是：`单文件`、`模块内多文件`、`跨模块多文件`、`跨系统多模块`、`无需修改`。
- `任务难度` 只能是：`简单`、`一般`、`困难`、`地狱`。
- `执行状态` 只能是：`待生成`、`已生成`、`已发送`、`Trae运行中`、`待验收`、`验收中`、`已完成`、`已跳过`、`超时待人工`、`失败`。它表示任务执行过程走到哪一步，不表示验收是否通过。
- `任务是否完成` 只能是：`已完成`、`未完成`。证据不足时也必须根据代码和可见结果判断为其中一个，不能写“暂时无法判定完成”。
- `过程与产物是否满意` 只能是：`满意`、`不满意`。只有关键目标完成且验收证据充分时才写 `满意`；只要主链路没验透、结果有明显缺口、或证据不够硬，就写 `不满意`。
- `Repo URL` 是同一个仓库共享的远程 git 地址；能识别到 `origin` 时应默认回填到该仓库所有行。固定格式必须是 Markdown 链接：`[owner/repo](https://github.com/owner/repo)`，例如 `[yuenlai/solo-6600007](https://github.com/yuenlai/solo-6600007)`。
- `Repo URL` 不能直接写 SSH 地址、裸 HTTPS 地址、带 `.git` 的地址或纯 `owner/repo`；这些格式都必须先转换成 `[owner/repo](https://github.com/owner/repo)` 再回填。
- `Commit ID` 是当前这一轮提示词对应的提交 ID；每一轮主提示词和修复提示词都要单独回填自己的值。
- `Commit ID` 只能以当前轮真实 commit 结果为准，不能口头记住待会再补；push 成功后必须立刻回填到当前轮这一行。

旧 Excel 不允许被误读执行。脚本发现旧列，如 `子文件夹名称`、`编号`、`状态`，必须报错并要求迁移或重新创建 v2 workbook。

常用 workbook 命令：

```bash
python3 $CODEX_HOME/skills/solo-faster/scripts/batch_prompt_workbook.py status --parent "<父目录>"
python3 $CODEX_HOME/skills/solo-faster/scripts/batch_prompt_workbook.py next --parent "<父目录>"
python3 $CODEX_HOME/skills/solo-faster/scripts/batch_prompt_workbook.py update --parent "<父目录>" --main 1 --round 1 --status 已生成 --prompt "<提示词>"
python3 $CODEX_HOME/skills/solo-faster/scripts/batch_prompt_workbook.py insert-fix --parent "<父目录>" --main 1 --prompt "<修复提示词>"
python3 $CODEX_HOME/skills/solo-faster/scripts/batch_prompt_workbook.py outcome --parent "<父目录>" --main 1 --round 1 --completed 未完成 --process-reason "<过程不满意原因>" --product-reason "<产物不满意原因>" --repo "<仓库路径>"
python3 $CODEX_HOME/skills/solo-faster/scripts/batch_prompt_workbook.py reason-check --parent "<父目录>" --main 1 --round 1 --process-reason "<过程不满意原因>" --product-reason "<产物不满意原因>"
python3 $CODEX_HOME/skills/solo-faster/scripts/batch_prompt_workbook.py finalize-round --parent "<父目录>" --repo "<仓库路径>" --main 1 --round 1
python3 $CODEX_HOME/skills/solo-faster/scripts/batch_prompt_workbook.py audit --parent "<父目录>" --repo "<仓库路径>"
```

## 批量生成规则

默认生成 43 条主提示词：

- `0-1代码生成`：20 条。
- `Feature迭代`：20 条。
- `代码理解`：1 条。
- `代码重构`：1 条。
- `工程化`：1 条。

默认编号顺序不要按任务类型整块堆放，必须尽量均匀穿插：

- 优先把数量最多的主类型交错排开，默认让 `0-1代码生成` 和 `Feature迭代` 尽量按 `0-1 / 迭代 / 0-1 / 迭代` 这种节奏穿插。
- `代码理解`、`代码重构`、`工程化`、`代码测试` 这类数量较少的类型，不要全部压到最后，应该尽量插入到整体序列中间，避免前面连续很多条都是同一类型。
- 自定义 `--task-counts` 时也保持同样原则：按编号执行时，相邻主提示词尽量不要是同一任务类型；数量少的类型优先打散插入，而不是集中成一段。
- 如果某个类型数量远多于其他类型，允许尾部出现少量连续项，但默认要先做到“前半段尽量均匀”，不要一开始就连续堆同类任务。

也支持手动指定主提示词的任务类型和条数：

- 使用 `scan --task-counts "任务类型=条数,任务类型=条数,..."`
- 允许的主提示词类型是：`0-1代码生成`、`Feature迭代`、`代码理解`、`代码重构`、`工程化`、`代码测试`
- 不允许把 `Bug修复` 放进 `--task-counts`
- 未出现在 `--task-counts` 里的主提示词类型默认按 `0` 条处理

难度模式：

- 默认 `中`。
- `低`：`简单`、`一般` 为主。
- `中`：`一般`、`困难` 为主。
- `高`：默认仍按 `困难` 生成，不自动批量写 `地狱`。

任务难度判断规则：

- 默认不要把任务难度判断为 `地狱`；能落到 `困难` 的一律写 `困难`。
- 只有用户明确要求极限难度，且任务同时具备跨系统、多端联动、高风险迁移或复杂架构重塑时，才允许少量写 `地狱`。
- 不能因为提示词看起来工作量大、文件多、业务描述长，就直接写 `地狱`。
- 自动 `scan` / 批量生成默认不产生 `地狱`；如果确实需要，必须人工逐条确认后再改。

生成前先识别仓库业务领域，同一仓库所有任务复用同一个业务领域。识别可以由脚本根据技术栈给出初值；如果明显不准，人工改成业务领域枚举中的一个。只要仓库同时存在明确前端信号和后端 / API / 数据库 / 服务端信号，业务领域必须判为 `全栈 Web 应用`，不能因为后端框架或服务端文件更多就写 `纯后端服务`。

每条主提示词写入：

- `主提示词编号` 从 `1` 开始连续编号。
- `轮次` 为 `1`。
- `行类型` 为 `主提示词`。
- `Repo ID` 为 `当前仓库目录名-主提示词编号`。
- `Trae Session ID` 先置空；提示词发送成功后，必须立刻复制并回填，不能拖到验收时再补。只要这一动作失败，就不能进入人工等待、验收、修复、提交或下一个主提示词。
- `执行状态` 从 `待生成` 更新为 `已生成`。

生成质量规则继续保留：

- 必须贴当前仓库，不能是任何项目都能套用的通用模板。
- 必须是用户可感知的问题或需求。
- 默认用中文。
- 不要出现文件路径、函数名、类名、实现指挥。
- 生成内容必须按仓库真实业务和代码能力取材，不能只套占位句。
- 如果任务类型是 `0-1代码生成`，提示词默认直接写“要新增的能力本身”，从新增能力起笔，优先写成 `新增xxx功能，……`。
- 不要用“现在只有……，希望新增……”这种把现状铺在前面的句式。
- 如果一句提示词删掉“目前 / 现在 / 已有 / 现有 / 原来”这类现状前缀后仍然成立，默认删掉再输出。
- 在现有项目里新增一块用户能独立感知的新能力、新入口或新流程，默认按 `0-1代码生成`。
- 在已有能力上继续补规则、补交互、补展示，或者在原流程上往前推一点，默认按 `Feature迭代`。
- 如果核心目标是做出一个原来没有的独立能力、入口或流程，即使挂在现有页面里，也优先按 `0-1代码生成`。

批量生成时要轻扫仓库：

- 优先用 `rg --files`、`git status --short`、`git diff --stat`。
- 只读和当前切入点直接相关的文件。
- 对同一任务类型做历史验重：

```bash
python3 $CODEX_HOME/skills/solo-faster/scripts/query_prompt_history.py --project-path "<仓库路径>" --task-type "<任务类型>"
```

生成完每条提示词后，运行输出级禁词检查，记录历史，并立刻写回 Excel。不要等全部主提示词生成完再统一写。

## Trae 串行执行状态机

执行顺序按 `主提示词编号` 升序，同一主提示词内按 `轮次` 升序。

主提示词任务开始前：

1. 打开目标仓库的 Trae 窗口。
2. 确认前台 Trae 窗口标题属于目标仓库。

更具体的默认动作：

1. 用仓库路径直接打开 Trae，例如：`open -a "/Applications/Trae CN.app" "<仓库路径>"`。
2. 如果 Trae 已经开着，也仍然要把目标仓库目录切到前台工作区，不要假定当前窗口就是对的项目。
3. 只有确认 Trae 当前窗口对应的是目标仓库后，才进入“请新建任务，并发送这条主提示词”这一步。

默认发送方式：

- 不再默认使用 `@电脑` / `computer-use` 去控制 Trae 的输入框、发送按钮、新建任务快捷键或窗口切换。
- 不再默认依赖 `send_batch_prompts_to_trae.py send` 或 `new-task` 去真实操作 Trae UI。
- 默认改为：Agent 负责根据 workbook 的断点续跑结果，明确告诉用户“现在该不该新建任务”“当前要发送哪一条提示词”“是否属于修复轮次且必须沿用当前任务上下文”；用户自己在 Trae 里完成发送。
- 用户发送完成后，会立刻回复当前轮自己的完整 `Trae Session ID`；Agent 收到后再回填 Excel，并把当前轮推进到 `Trae运行中`。
- 无论是用户手动发送，还是显式启用 `--auto-ui` 自动发送，发送刚完成时都只允许先写成 `已发送`；只有当前轮自己的完整 `Trae Session ID` 已经回填成功后，才能推进到 `Trae运行中`。
- 从当前版本开始，`send_batch_prompts_to_trae.py send` 默认进入低 CPU 的人工发送模式，只输出发送说明与提示词，不再自动切窗口、截图、点输入框或回车发送。只有显式传入 `--auto-ui` 时，才允许走 Trae UI 自动化。
- 从当前版本开始，`send_batch_prompts_to_trae.py monitor` 默认进入低 CPU 的人工等待模式，只返回当前待观察行和后续动作，不再持续轮询 Trae 完成态。只有显式传入 `--active-monitor` 时，才允许启用截图分析、日志追踪和轮询。
- `send_batch_prompts_to_trae.py` 保留给 dry-run、自检、锁管理和少量状态辅助，不再是默认真实发送路径。

发送辅助命令：

```bash
python3 $CODEX_HOME/skills/solo-faster/scripts/send_batch_prompts_to_trae.py send --parent "<父目录>" --repo "<仓库路径>" --range "1" --dry-run
```

低 CPU 模式下，如需只拿“当前应该发什么”的说明，可直接运行：

```bash
python3 $CODEX_HOME/skills/solo-faster/scripts/send_batch_prompts_to_trae.py send --parent "<父目录>" --repo "<仓库路径>" --range "1"
```

只有在确实需要恢复旧的自动化发送行为时，才显式加：

```bash
python3 $CODEX_HOME/skills/solo-faster/scripts/send_batch_prompts_to_trae.py send --parent "<父目录>" --repo "<仓库路径>" --range "1" --auto-ui
```

默认人工发送规则：

- Agent 每次只告诉用户当前下一步该发什么，不替用户真实点击 Trae。
- 默认必须服从 `next` 的断点续跑结果；存在同一主提示词的修复轮待发送时，不能跳到后面的主提示词。
- 发送提示词给用户时，`【第 X 个主提示词，第 Y 轮】` 是最高优先级强制标签：只要需要用户发送新提示词，回复第一行必须以这个标签开头，不能省略，不能放到正文后面，不能只输出提示词正文。
- 这个强制标签属于发送说明，不属于提示词正文，不写入 Excel 的 `提示词` 字段，也不受“输出禁令”里的轮次词限制。
- `【第 X 个主提示词，第 Y 轮】` 必须明确写出“实际执行到的第几个主提示词”和轮次，并用 `【】` 包起来，格式固定为：`【第 X 个主提示词，第 Y 轮】`。
- 这里的 `第 X 个主提示词` 只能按当前 workbook 中 `行类型=主提示词` 的实际执行顺序计算；不能用 `主提示词编号` 代替，也不能用整张表里的第几行代替。
- 如果下一条是新的主提示词第 1 轮，Agent 必须明确告诉用户：`【第 X 个主提示词，第 1 轮】【需要新建任务】`，然后给出该条提示词正文。
- 如果下一条是第 2 或第 3 轮修复提示词，Agent 必须明确告诉用户：`【第 X 个主提示词，第 Y 轮】【不要新建任务】`，然后给出该条修复提示词正文。
- Agent 给用户的发送说明必须足够短，默认只包含三件事：`【第 X 个主提示词，第 Y 轮】`、是否新建任务、提示词正文。
- 上一条里的 `【第 X 个主提示词，第 Y 轮】` 不能再缩写成别的口径，也不要省略“主提示词”三个字，避免和整表第几条、修复提示词第几条混淆。
- 发送说明里的“修复提示词 / 主提示词”这类流程标记只允许出现在说明头里，不能混进提示词正文，更不能在正文里写“上一轮 / 这一轮 / 第几轮 / 首轮 / 次轮 / 本轮 / 当前轮 / 第二轮 / 第三轮”等轮次承接词。
- 用户自己发送完成后，必须立刻回复当前轮自己的完整 `Trae Session ID`。
- 只有在用户已经回复完整 `Trae Session ID` 后，Agent 才能把当前轮写成 `Trae运行中`。
- 如果走 `--auto-ui` 自动发送，脚本也只能把当前轮停在 `已发送` 并提示立刻回填 `Trae Session ID`，不能自行跳过这个步骤直接写成 `Trae运行中`。
- 如果用户没有回 `Trae Session ID`，或回的值明显不是当前轮自己的完整值，流程必须暂停；不能继续人工等待、验收、修复、提交或下一条发送。
- 每一行状态变化都必须立即写回 Excel，并且要把当前时点已经确定的相关字段一起补齐；禁止只更新 `执行状态`，却漏掉同一步已经拿到的 `Trae Session ID`、`备注`、`Commit ID` 等信息。

断点续跑判断：

- `已生成`：告诉用户当前要发送的提示词。
- `已发送` / `Trae运行中`：先确认当前轮 `Trae Session ID` 已经回填；回填完成后等待用户人工观察 Trae。用户回复“继续”或明确说明当前轮已跑完后，再进入验收；如果 `Trae Session ID` 为空，优先向用户索取，补不齐则暂停。
- `待验收`：进入验收。
- `验收中`：恢复验收或暂停确认。
- `已完成`：该主提示词不重复执行。
- `超时待人工`：等待用户检查 Trae；用户回复继续后恢复后续流程。
- `失败`：暂停确认。
- Excel 状态和 git / Trae 实际状态冲突时暂停确认。

## 人工等待与继续

用户发送成功并回填 `Trae Session ID` 后，不再自动监控 Trae，不再默认每 2 分钟轮询，也不再要求脚本持续输出监控结果。当前标准流程改为人工等待：

- 发送完成后，把当前轮保持在 `Trae运行中`。
- 由用户自己在 Trae 窗口里观察运行状态。
- 用户确认当前轮已经跑完后，回复“继续”或明确说明“第 X 个主提示词，第 Y 轮已完成，可以继续”。
- 收到用户继续指令后，再进入验收，并把当前轮状态推进到 `待验收` / `验收中`。
- 如果用户确认 Trae 还没跑完，就继续等待，不做自动轮询。
- 如果用户确认 Trae 卡住、失败或需要介入，则按实际情况写 `超时待人工` 或 `失败`，等待处理后再继续。

保留 `monitor` 命令仅用于脚本自检或手工调试，不再属于默认执行路径；当前默认批量流程不依赖锁。

如果只是恢复断点续跑并确认“哪些轮次还在 `Trae运行中`”，不要开主动监控，直接运行：

```bash
python3 $CODEX_HOME/skills/solo-faster/scripts/send_batch_prompts_to_trae.py monitor --parent "<父目录>" --repo "<仓库路径>"
```

只有在明确接受更高 CPU 和更重的 Trae UI 检测开销时，才显式加：

```bash
python3 $CODEX_HOME/skills/solo-faster/scripts/send_batch_prompts_to_trae.py monitor --parent "<父目录>" --repo "<仓库路径>" --active-monitor
```

## 验收与回填

验收开始时先写 `验收中`。如果当前轮的 `Trae Session ID` 还没在发送后成功回填，先补齐；补不齐就停止，不允许继续验收。只有用户已经明确回复“继续”或等价确认当前轮已跑完后，才能进入验收。验收结束必须把 `执行状态` 写成 `已完成`，再根据结果把 `任务是否完成` 写成 `已完成` 或 `未完成`，同时回填 `修改范围`、满意状态和原因字段。无论当前轮验收结果是 `已完成` 还是 `未完成`，都必须继续尝试当前轮 git 提交流程；该轮验收结束后必须把当前轮代码提交并推送到远程 git，并把当前轮对应的 `Commit ID` 回填到 Excel。终态行必须有 `修改范围`，不能留空。

补充硬规则：

- 验收完成不等于当前轮完成；只有 Excel 里当前轮应有字段都已经写齐，当前轮才算真正收尾。
- git push 成功后，必须先回填当前轮 `Commit ID`，再执行 `next` 或告诉用户发送下一条提示词；这个顺序不能颠倒。
- 如果发现历史轮次有字段漏填，优先补齐漏项，再继续当前流程。

默认裁决原则：

- 验收默认从严，不做“差不多可以”的宽松放行。
- 只要关键主链路里有一个核心点没有被证实，就默认判 `未完成`。
- 只看到了代码改动，但没看到关键结果真的出现，默认偏 `未完成`，不要偏 `已完成`。
- 只验证了静态渲染，没验证关键交互、关键状态更新、关键跳转、关键保存结果时，默认偏 `未完成`。
- 前端任务里，如果 prompt 明确包含弹窗、表单、保存、列表刷新、路由跳转、筛选、分组、统计、图表、联动、回显、删除、编辑、权限、分页、拖拽、补签、回看等动作词，至少要实际验到对应动作生效；缺任意主链路证据都默认 `未完成`。
- “满意”的门槛比“已完成”更高：不仅要完成主链路，还要没有明显漏验、没有明显副作用、没有容易一眼发现的体验缺口。
- 如果你在“已完成”和“未完成”之间犹豫，默认先判 `未完成`，给出修复提示词，不要为了推进节奏而放宽。

`Trae Session ID` 回填规则：

- 当前轮提示词由用户发送成功后，就要立刻把 `Trae Session ID` 回给 Agent，不要等到验收时才补。
- 完整 `Trae Session ID` 只认用户从 UI 实际复制出来并发回的值。
- Trae 本地 `ai-agent` 日志只能用于辅助确认“提示词已发送”以及“剪贴板里主体字段是否和当前消息一致”，不能用于生成、补齐或推断完整 `Trae Session ID`，尤其不能用来决定尾部时间。
- 如果没有拿到用户从 UI 实际复制回来的值，即使日志里已经出现 `trace_id`、`message_id`、`user_message_id`，也仍然视为当前轮 `Trae Session ID` 未回填成功；必须继续向用户索取或暂停人工处理。
- 只要当前轮 `Trae Session ID` 没复制成功，这一轮就不能视为有效执行；后续步骤一律暂停。
- 主提示词第 1 轮有自己的 `Trae Session ID`。
- 每一条修复提示词轮次也有自己独立的 `Trae Session ID`。
- 不能把上一轮、同一主提示词的别的轮次、或当前任务里别的消息的 `Trae Session ID` 复用到当前轮。
- 同一主提示词下的所有轮次都必须共用同一个 `Repo ID`，不能因为修复轮次而变更。

git 回填规则：

- `Repo URL` 对同一个仓库内所有行保持一致，优先取 `git remote get-url origin`，回填前统一转换为 `[owner/repo](https://github.com/owner/repo)`。
- 如果远端是 `git@github.com:owner/repo.git`、`https://github.com/owner/repo.git`、`https://github.com/owner/repo` 或 `owner/repo`，Excel 里最终都只能写 `[owner/repo](https://github.com/owner/repo)`。
- 真实执行时默认使用 GitHub SSH 远端；如果 `origin` 还是 `https://github.com/...`，先切到对应的 `git@github.com:owner/repo.git`，再继续后面的 push。
- 批量流程默认假设本机已经配置好可用的 SSH key 与 GitHub 授权；不要把 HTTPS 用户名密码交互留到执行中途再处理。
- `Commit ID` 必须对应“当前这一轮提示词验收结束后提交并推送出来的那一个 commit”。
- `Commit ID` 必须是 40 位完整 git SHA，不能写短 SHA、日志片段、人工拼接值或推断值。
- 当前轮产生代码改动并执行 git 提交时，提交 message 必须严格等于当前轮自己的 `Trae Session ID`。
- 主提示词第 1 轮和每一条修复提示词轮次都各自有独立的 `Commit ID`，不能复用上一轮。
- 如果当前轮没有产生代码改动，`Commit ID` 保持为空，并且必须明确提示用户手动确认是否继续；在用户确认前，不得自动继续后面的主提示词或修复轮次。
- 只要当前轮已经成功提交并推送，就不允许出现 `Commit ID` 仍为空的情况；若为空，必须视为回填遗漏并立即补齐。
- 每次执行 `next` 或发送下一条提示词前，先跑 `audit`；只要审计失败，就先修 Excel 或 git 状态，不能继续推进。

验收优先级：

- 先看 git 基线后的代码改动，再决定是否运行命令、启动服务或使用浏览器。
- 如果通过代码变更已经能明确判断未完成，可以不启动浏览器，直接判 `未完成`。
- 如果通过代码变更只能推测“可能完成”，不能直接判 `已完成`；这时必须补足运行证据或页面证据。
- 纯后端、CLI、脚本、数据处理项目不默认启动浏览器；优先通过代码、接口逻辑、数据流、测试或命令行结果判断。
- 前端强交互、路由、筛选分页、列表详情联动、多步骤表单、拖拽、缩放、滚动、画布、3D 或可视化任务，在代码证据不足时要实际验证。
- 只要进入页面验收，默认必须使用 Codex 内置浏览器 `@browser` / `@浏览器`，不要优先用外部系统浏览器。
- 本地项目页面验收时，优先在 Codex 内置浏览器中打开 `localhost`、`127.0.0.1`、`::1` 或明确的本地预览地址；需要点击、输入、切换路由、触发表单、查看渲染结果时，都在内置浏览器里完成。
- 如果当前任务已经在 Codex 内置浏览器里打开了目标页面，后续验收优先复用这个浏览器上下文，不重复换到别的浏览器。
- 验收时说的“内置浏览器”固定指 Codex app 自带的 in-app Browser，也就是用户口中的 `@浏览器`；不要误解成系统默认浏览器、Trae 内预览页或其他外部浏览器窗口。
- 只有当内置浏览器确实无法覆盖某个必须的交互，或用户明确要求别的浏览器时，才允许降级到其他方式；如果降级了，必须在验收结论里说明原因。
- 构建失败、核心测试失败、项目无法启动且代码无法证明核心行为完成，默认 `未完成`。
- 只通过 `npm run build`、静态编译通过、类型检查通过，不能直接推出功能完成；这些只能作为基础通过证据，不能替代主链路验收。
- 如果页面任务涉及“新增入口”，必须验到入口能找到、能点进去、并能通往目标功能。
- 如果页面任务涉及“新增表单/弹窗”，必须验到打开、输入、提交、关闭后的结果变化，不能只验打开。
- 如果页面任务涉及“保存/创建/编辑/删除”，必须验到提交后的持久结果变化，例如列表、详情、统计、徽标、卡片、回显或刷新后仍存在。
- 如果页面任务涉及“分类/筛选/分组/统计/图表”，必须至少验两组不同数据下结果确实变化，不能只看单一默认态。
- 如果页面任务涉及“详情页回看/历史记录”，必须验到至少一条新数据或目标数据真实出现在时间序列里，并且顺序、入口或内容符合预期。
- 如果页面任务涉及多个界面联动，至少抽查两个界面都已同步；只改了一个界面且另一个关键界面没验到，默认 `未完成`。
- 修改范围指的是“当前这一条提示词对应的代码改动范围”，不是整个仓库历史累计范围。判断时以当前轮验收看到的改动为准，不把 Excel 文件和 `.DS_Store` 算进修改范围。
- 修改范围规则：单个文件为 `单文件`；同一模块内多个文件为 `模块内多文件`；多个普通模块为 `跨模块多文件`；涉及多个端或系统边界时必须写 `跨系统多模块`，例如用户端和后台端同时改动、前端和后端代码同时改动、API / 数据库 / 后台管理与页面样式或用户页面同时改动。
- `Web 前端` 仓库里，页面、组件、样式、静态资源、前端状态、路由、构建配置和包配置之间的多文件改动默认仍属于前端内部多模块，不能仅因为文件分散就写 `跨系统多模块`；只有同时触及后端、API、数据库、后台管理、服务端脚本等系统边界时才写 `跨系统多模块`。

前端任务最小验收清单：

- 先列出这条 prompt 的 2 到 4 个关键承诺点，再逐个找证据，不要看个大概就下结论。
- 至少覆盖 1 条主入口链路。
- 至少覆盖 1 条保存后结果链路。
- 至少覆盖 1 条跨页面或跨区域结果回看链路；如果任务本身没有跨页面要求，才可以省掉这一条。
- 至少确认 1 个失败风险最高的边界点，例如空态、重复操作、刷新后是否保留、编辑后是否回显、删除后是否消失、统计是否同步。
- 任意一条没验到，就不要写 `满意`。

过程验收触发条件：

- 只有用户这次消息里真的贴了 Trae 过程信息，才叠加做过程验收。
- 如果用户没贴 Trae 过程信息，不要凭空补过程细节；但只要结果不满意，仍然要从当前结果里反推一句合规的过程不满意原因。
- 如果结果没问题，但过程里明显有冗余思考、待办拆法很怪、语言跑偏、一直重复解释、执行节奏很乱，也不能直接算高质量完成。
- 过程验收重点看理解 prompt、目标是否抓偏、推理是否精简、输出是否正常、任务规划是否合理、工具使用是否高效、操作是否有边界感、总结是否正确、是否虚假完成。

严格输出要求：

- 验收结论不能只写“已完成”或“未完成”。
- 在内部判断时，必须先列出“本次实际验了什么”“还没验到什么”“为什么这些缺口足以判未完成或不满意”，再回填 Excel。
- 如果最终仍判 `已完成`，心里必须能回答清楚：主入口在哪、核心操作是什么、结果出现在哪、是否真的看到了结果、有没有至少一个副作用或联动点被抽查过。
- 只要这些问题里有一个回答不硬，就默认改判 `未完成`。
- 给用户的验收结果必须简洁，不展开内部推理，只回传关键回填结果和需要用户处理的点。
- 任何异常、阻塞、字段缺失、git push 失败、需要用户确认的地方，都必须用 `【】` 包裹并放在回复前半段。
- 验收收尾回复必须包含：`是否完成`、`过程与产物是否满意`、`修改范围`、`不满意原因`、`Commit ID`、`远程推送`、`Excel 回填`。其中满意或已完成时，`不满意原因` 写 `无`。

回填命令示例：

```bash
python3 $CODEX_HOME/skills/solo-faster/scripts/batch_prompt_workbook.py outcome --parent "<父目录>" --main 1 --round 1 --completed 已完成 --repo "<仓库路径>"
```

未完成时只允许把原因写入 `不满意原因`。如果历史 workbook 仍带 `过程不满意原因` 和 `产物不满意原因` 两列，也必须保持为空，不能再分列保存。

`不满意原因` 必须由过程原因和产物原因合并生成，格式固定为：

```text
过程不满意：xxx。
产物不满意：xxx。
```

不允许写入“暂时无法判定完成”。不满意原因不能写“只通过代码判断”“暂时无法判断”“浏览器没验证”等验收背景。

## 修复轮次

每个主提示词最多执行 3 轮：

- 第 1 轮是主提示词。
- 第 2、3 轮是修复提示词。
- 任意轮次验收为 `已完成`，该主提示词任务结束。
- 第 3 轮仍 `未完成`，不再继续修复。

修复提示词规则：

- 沿用同一个 `主提示词编号`。
- `轮次` 为 `2` 或 `3`。
- `行类型` 为 `修复提示词`。
- `Repo ID` 沿用该主提示词第 1 轮的值，不重新生成。
- `任务类型` 固定为 `Bug修复`。
- 只写已经验出来的问题。
- 不要把最初大需求整段重写一遍。
- 默认写成“修复……问题……”。
- 修复提示词正文直接从 `修复` 开始，不加“继续在当前任务中”“基于前一轮”“上一轮存在问题”等上下文说明。
- 直接描述要补的行为结果，不写过程指挥、文件路径、函数名、类名、按钮名或日志原文。
- 发给用户前和写入 Excel 前都要做输出级禁词检查。
- 插入修复提示词后，断点续跑的下一步必须是同一主提示词的修复轮次，不能跳到下一个主提示词。
- 修复轮次必须继续使用当前 Trae 任务上下文发送，不新建 Trae 任务；因为修复提示词依赖同一个上下文中的已验问题。

插入修复行：

```bash
python3 $CODEX_HOME/skills/solo-faster/scripts/batch_prompt_workbook.py insert-fix --parent "<父目录>" --main 1 --prompt "<修复提示词>"
```

## Bug 修复上限

每个仓库允许生成的 `Bug修复` 行数上限，不再固定写死为 10 条，而是不能超过当前这个 Excel 里 `0-1代码生成` 主提示词的总数。主提示词第 1 轮不计入 Bug 修复数。

- 只统计当前 workbook 中 `行类型=主提示词` 且 `任务类型=0-1代码生成` 的总数，作为该仓库 `Bug修复` 的动态上限。
- 当 `Bug修复` 行数已经与这个 `0-1代码生成` 总数相同，后续所有验收即便结果是 `未完成`，也不再生成下一轮修复提示词。
- 达到这个上限后，当前轮仍要先按规则完成 Excel 回填和 git 提交流程，然后直接进入下一个主提示词。
- 已达到上限时，断点续跑不能再执行新的 `insert-fix`；下一步必须是下一个主提示词。

## 多轮次数量上限

单个仓库内，不是所有主提示词都允许进入修复轮次。除总的 `Bug修复` 行数上限外，还要额外控制“进入多轮次的主提示词数量”：

- `0-1代码生成`：最多只允许 5 个主提示词进入多轮次。
- `Feature迭代`：最多只允许 5 个主提示词进入多轮次。

这里的“进入多轮次”指的是：某个主提示词已经生成过至少 1 条 `Bug修复` 修复轮次，也就是它已经从第 1 轮进入了第 2 或第 3 轮。

规则：

- 一个 `0-1代码生成` 主提示词只要生成了第 2 轮修复提示词，就计入 `0-1代码生成` 的多轮次名额。
- 一个 `Feature迭代` 主提示词只要生成了第 2 轮修复提示词，就计入 `Feature迭代` 的多轮次名额。
- 已经进入多轮次的主提示词，可以在自己的上下文里继续跑到第 3 轮；不会因为名额随后用满，就把已经开始的多轮次强行截断。
- 但新的主提示词如果还没进入修复轮次，而所属任务类型的多轮次名额已经用满，那么后续验收即便结果是 `未完成`，也不再生成该主提示词的下一轮修复提示词。
- 这时仍然要先完成当前轮 Excel 回填和 git 提交流程，然后直接进入下一个主提示词。
- 断点续跑和 `insert-fix` 都必须同时满足两层约束：`Bug修复` 总量上限，以及所属任务类型的多轮次数量上限。

## Git 提交与推送

不再要求每个主提示词结束后把仓库回退到执行前状态。改为每执行完一个轮次提示词，包括修复轮次，都要提交并推送当前轮代码。

要求：

- `Repo URL` 写入 Excel，对同一仓库所有行保持一致，且必须使用 `[owner/repo](https://github.com/owner/repo)` 格式；任何其它 GitHub 远端格式都要先转换。
- 推送远端默认使用 SSH；如果仓库 `origin` 还是 GitHub HTTPS 地址，先改成对应 SSH 地址，再执行 `git push origin HEAD`。
- 无论当前轮验收结果是 `已完成` 还是 `未完成`，都必须尝试执行当前轮 git 提交流程。
- 当前轮产生代码改动时，必须提交并推送远程 git。
- 当前轮 git 提交的 message 必须严格使用该轮已经回填好的 `Trae Session ID`，不能自拟别的提交说明。
- 提交完成后，把当前轮对应的 `Commit ID` 回填到 Excel。
- `Commit ID` 必须是一轮一值；主提示词和修复轮次分别记录自己的提交结果。
- 如果当前轮没有代码改动，不强造 commit，`Commit ID` 保持为空，并在备注里说明，同时必须明确提示用户手动确认是否继续；在收到确认前，流程暂停。
- 如果 git 提交或推送失败，则当前轮写 `失败` 并暂停，必须明确提示用户这一步需要人工处理，不能假定远程已经同步，也不能自动继续后续轮次。

git 与 Excel 的固定顺序：

1. 完成当前轮验收回填。
2. 执行 `finalize-round`。
3. `finalize-round` 内部用当前轮 `Trae Session ID` 创建 git commit。
4. `finalize-round` 内部执行 git push。
5. `finalize-round` 内部读取 40 位真实 `Commit ID` 并立刻写回 Excel。
6. 执行 `audit` 并确认通过后，才能执行 `next` 或通知用户下一步。

第 2 到 6 步是强制收尾步骤，不能省略，不能延后，不能靠人工记忆补救。除非明确是脚本自检，真实流程不要手写 git commit / git push 后再人工回填。

## Trae 新任务规则

只有当前主提示词已完成、达到第 3 轮仍未完成、或 Bug 修复数已达到当前 workbook 的动态上限而不再修复时，才算该主提示词任务结束。进入下一个主提示词前，必须由用户自己在 Trae 窗口新建任务。

同一主提示词的第 2、3 轮修复提示词不新建任务，必须在当前 Trae 任务里继续发送。

默认流程中不再要求 Agent 自动触发 `control+command+n` 或调用 `new-task` 命令。改为：

- Agent 判断当前下一步是否需要新建任务。
- 如果需要，明确告诉用户：`请先新建任务，再发送下面这条主提示词。`
- 如果不需要，明确告诉用户：`不要新建任务，直接在当前任务下发送下面这条修复提示词。`
- 用户自己完成新建任务与发送。
- 用户发完后立刻把当前轮 `Trae Session ID` 回复给 Agent。

## 断点续跑

如果 Excel 已存在，先读取当前状态，不重复创建或覆盖。使用：

```bash
python3 $CODEX_HOME/skills/solo-faster/scripts/batch_prompt_workbook.py next --parent "<父目录>"
```

续跑规则：

- 已完成的主提示词不重复执行。
- 已生成未发送的继续提示用户发送。
- 已发送或 `Trae运行中` 的先确认 Trae 状态。
- `待验收` 进入验收。
- `验收中` 恢复或提示确认。
- 未完成但未达 3 轮且 Bug 修复数未达到当前 workbook 动态上限的继续修复；如果修复提示词已经生成，下一步必须发送该修复提示词。
- 达到 3 轮或 Bug 修复数已达到当前 workbook 的动态上限的进入下一个主提示词。
- 如果某一行虽然状态已经推进，但当前状态应有的关键字段还没补齐，例如 `Trae Session ID`、`Commit ID`、`修改范围` 等，`next` 必须先返回暂停确认，不能继续续跑。
- Excel 状态和 git / Trae 实际状态冲突时暂停确认。

## 输出禁令

只要是最终要写入 Excel、发给用户、或可能被复制使用的内容，都必须先过这一层。

例外：当需要用户发送新提示词时，发送说明第一行里的 `【第 X 个主提示词，第 Y 轮】` 是强制标签，不属于提示词正文，不写入 Excel，不受本节轮次禁词限制。这个例外只允许出现在发送说明头部，提示词正文、`不满意原因`、过程原因、产物原因仍然严格禁用轮次承接词。

输出级内容包括：

- 生成的提示词。
- `不满意原因`。
- 用来合并生成 `不满意原因` 的过程原因和产物原因。
- 跟这几类内容紧挨着的简短结论补充。

这些内容里严禁出现：

- `稳定`
- `收口`
- `收住`
- `落下来`
- `这一轮`
- `上一轮`
- `下一轮`
- `本轮`
- `当前轮`
- `当前轮次`
- `首轮`
- `次轮`
- `第几轮`
- `第一轮`
- `第二轮`
- `第三轮`
- `同上`
- `（继续）`

这里要特别强调：生成的提示词、`不满意原因`、过程原因、产物原因里，不能出现任何轮次相关词汇或跨次承接话术。除了上面列出的禁词，也包括同义变体，例如“前一轮”“后一轮”“第2轮”“第3轮”“上个版本那次修改”这类表达。

禁令不只是禁掉词本身，也禁掉带前缀、带括号、带引用感的写法。表达“基于当前结果继续修”时，直接写当前看到的问题和要补的行为，不写跨次承接词。

格式硬规则：

- `提示词`、`不满意原因`、用来合并生成 `不满意原因` 的过程原因和产物原因都必须以标点结尾，优先用 `。`、`！`、`？`。
- 过程原因不能以页面、按钮、文案、提示或内容显示英文作为理由；英文显示属于产物现象，只能放进产物原因。

## 不满意原因规则

任务未完成时必须写一个合并后的 `不满意原因`；如果历史 workbook 仍带旧列，也必须清空 `过程不满意原因`、`产物不满意原因`。

生成阶段硬规则：

- 未完成行的 `不满意原因` 是必填项，不能因为一次候选原因命中规则就留空、跳过或改成满意；必须根据真实验收缺口重新生成候选原因。
- 先根据当前任务的真实验收缺口写候选过程原因和产物原因，再用 `reason-check` 做写入前自检；自检失败或返回 `regenerate_required=true` 时，必须重写候选原因并再次运行 `reason-check`，循环到 `ok=true` 后才能写入。
- `outcome` / `update` 写入不满意原因时会做同样的前置校验；如果和历史原因长片段雷同、trigram 相似过高、分段复读或贴着提示词复述，命令会直接失败。
- 写原因时先看同一 workbook 里已有 `不满意原因`，主动避开相同句式、相同语序和相同产物段，不要等后置审核发现。
- `outcome` 或 `update` 因不满意原因校验失败退出时，不能继续提交、推送、发送下一条或把该行终结；必须回到原因生成步骤，改写到通过为止。

合并格式固定为两行：

```text
过程不满意：xxx。
产物不满意：xxx。
```

质检硬性要求：

- `过程不满意` 和 `产物不满意` 都必须分析，分别说清过程哪里差、产物哪里差；不能只写其中一边。
- 原因必须定位到具体环节和步骤：写清问题发生在规划、读取、编辑、命令执行、测试、验收、提交等哪个环节；如果用户提供了 Trae 过程或工具记录，要指出第几步、哪个工具调用、哪次文件改动。
- 必须用证据，不用形容词：优先引用客观事实，如报错信息、失败命令、出错文件名、函数名、漏掉的具体需求、未实现的交付项、未运行的验证步骤。
- 禁止只写“效果不好”“不行”“一般”“没完成任务”“代码有 bug”“过程比较乱”“需求实现不完整”等笼统表面症状。
- 禁止把验收者视角当原因：不能写“我无法看到”“Codex 内看不到”“缺少浏览器佐证”“暂时不能证明”“还不能按验收标准确认”“构建通过但缺页面结果佐证”等。
- 如果真实问题是没有运行页面或没有验到交互，必须写成模型行为问题，例如“模型在完成判断前没有创建多条计时器数据并验证排序展示”，以及产物侧风险“TimerList 的排序逻辑缺少运行中与即将结束两类计时器同时存在时的展示证据”，不能写成验收者看不到。
- 只能写由模型本身能力、规划、理解、实现、验证或工具使用造成的问题；不能把环境、网络波动、模型请求失败、依赖源临时不可用等外部原因写成不满意原因。
- 如果确实有环境错误，也只能写模型没有识别、没有绕开、没有给出合理降级验证或错误处理方案，不能把“环境坏了”本身当成原因。

三要素写法：

- 每个问题点都尽量包含触发节点（When）、实际行为（What）、业务影响（Impact）。
- 更好的原因还要补出根因和正确做法，例如模型为什么会错，合理做法应该是什么。
- 一条数据可能有多个问题点；每个问题点都要按三要素展开，不要求句式完全一致，但不能缺证据和影响。

过程问题推荐写法：

```text
在【第几步/哪个环节】，模型【做了什么具体行为】→ 导致【什么后果】（→ 【根因和正确做法】）。
```

产物问题推荐写法：

```text
产物在【哪个文件/功能】存在【什么具体问题】，表现为【客观证据，如报错/缺失需求】，无法满足【哪条需求】。
```

过程与产物闭环写法：

```text
在【第几步/哪个环节】，模型【做了什么具体行为】→ 导致产物在【哪个文件/功能】存在【什么具体问题】，表现为【客观证据，如报错/缺失需求】，无法满足【哪条需求】（→ 【根因和正确做法】）。
```

如果审核意见指出 `User Prompt` 同义改写、模板换皮或提示词雷同，但当前任务只允许整改不满意原因，绝对不能为了过审改 `提示词` / `User Prompt`；只能修正不满意原因本身，提示词问题必须按提示词治理流程另行处理。

过程原因：

- 默认只写一个最核心的问题点，优先压成一句话。
- 只描述当前暴露出来的问题，不带基线、端口、浏览器等验收背景；但如果用户提供了 Trae 过程、工具调用或命令记录，必须写清具体步骤、工具、命令或文件改动证据。
- 不允许用第一人称验收视角写原因，例如“我无法看到”“我不能确认”“我没法证明”；要改写成模型漏做的验证动作、漏覆盖的场景或产物缺陷证据。
- 不允许用“上一轮少验了”“这一轮做偏了”“第 2 轮没有补到位”这类带轮次承接的句式。
- 当问题和代码证据有关时，允许并鼓励写具体文件名、函数名、字段名、失败命令或报错片段；不要为了自然语言化而抹掉质检需要的证据。
- 可以写模型行为问题：理解偏了、目标抓偏了、任务拆分漏主链路、验证顺序不对、过早下完成判断。
- 也可以写更底层业务 / 代码结构根因：数据源口径不统一、派生指标消费链没建立、写读模型冲突、业务对象生命周期拆裂、唯一记录约束缺失、结果归属口径不统一。
- 不要只复述 Bug 现象。
- 不要写“没有处理好”“缺少校验”“没有核对结果”“实现不完整”等空话。
- 不要把页面英文、按钮英文、提示英文、内容英文作为过程理由。
- 不要在多个任务之间套用同一个过程段骨架后只替换业务名词。每条过程原因都要写出本任务独有的判断错误、链路遗漏、数据口径问题或验证顺序问题。

产物原因：

- 只写用户能看到、点到、感受到的问题现象。
- 不允许写“这一轮做完后还是有问题”“第二轮提交后入口仍不可用”这类轮次关联表述。
- 不允许把“构建通过”“缺少页面佐证”“暂时不能证明”写成产物原因；构建结果只能是验证背景，不是产物缺陷。产物原因必须写具体功能、文件、排序规则、状态、数据或交互缺陷。
- 尽量用自然语言描述业务影响，但必须保留足以定位问题的文件、接口名、字段名、数据表名、函数名、报错或缺失需求证据。
- 不夹带“和前一次相比怎样”这类轮次背景；可以写代码改错了什么、漏改了什么、工具或命令暴露了什么错误，只要这些是模型行为和产物缺陷的证据。
- 不写端口、浏览器、基线等背景。
- 不默认写“暂没法判断”，只有产物侧确实还无法直接判断时才使用，并且仍要以标点结尾。
- 同一批次内产物段不能复用 18 字以上连续片段，也不能把同一句“页面仍没有新增结果 / 用户无法持续看到结果”换业务名词后反复使用；必须换成该任务真实的入口、动作、结果位置和用户看不到的具体反馈。
- 不满意原因不能贴着提示词复述需求。提示词写“要做什么”，不满意原因必须写“为什么这次交付没有满足它”：过程段讲失误根因，产物段讲用户可见缺口。

反复打回的审核规则要点：

- 规则 29：不满意原因模板骨架雷同。禁止多行共享同一套“过程不满意：没有把 X 和 Y 当成主链路；产物不满意：用户无法看到 A、B、C”的句式。
- 规则 36：过程段或产物段单独复读。即使整段相似率不高，只要某一段高度相似，也视为复读。
- 规则 37：跨记录连续长片段雷同。不同记录的不满意原因不能出现 18 字以上一字不差的业务描述片段。
- 规则 38：提示词同义改写属于提示词问题，不能通过改不满意原因掩盖；在“不改提示词”的任务里只处理原因列。

更合适的例子：

- `过程不满意：任务拆分时只盯住完成动作本身，没有把打卡后的统计卡片和连续天数当成同一个即时反馈主链路一起验。`
- `产物不满意：在习惯打卡页点完成以后，今日完成、当前连续、最长连续和本周完成率还停在旧值，用户当场看不到打卡结果。`
- `过程不满意：数据设计上没有先明确日期计划只能有一个当天真实来源，空计划和带时段计划被拆成两套记录后，读取口径从一开始就偏了。`
- `产物不满意：用户创建第一个专注时段后，弹窗关闭但页面仍显示暂无专注计划，刚填写的时间安排没有出现在当天计划里。`

明确反例：

- 不要写 `过程不满意：这次改动集中在计时器列表的排序规则上，但我无法在 Codex 内直接看到多个计时器同时存在时的页面排布变化，因此还不能按你的验收标准确认用户真的更不容易盯错计时器。`
- 不要写 `产物不满意：实现中已经把运行中的计时器和即将结束的计时器往前排，构建也顺利通过，不过缺少应用内浏览器中的页面结果佐证，暂时不能证明这种优先级调整已经准确落到实际展示顺序。`
- 不要写 `没完成任务，不满意。`
- 不要写 `代码有 bug。`
- 不要写 `过程比较乱。`
- 不要写 `效果还行但不满意。`
- 不要写 `文件没改成功，一直在报错。`
- 不要写 `写的代码根本不符合我的需求。`
- 不要写 `越改问题越多。`
- 不要写 `需求实现得不完整。`
- 不要写 `过程不满意：页面里还有英文文案。`
- 不要写 `过程不满意：按钮显示成英文。`
- 不要写 `实现上少了校验。`
- 不要写 `没有核对结果。`
- 不要写 `没有处理好。`
- 不要写 `实现不完整。`

合格示例：

- `过程不满意：在完成判断前，模型只依据计时器排序代码和构建结果下结论，没有创建多个运行中、即将结束和普通计时器同时存在的数据场景来验证列表顺序，漏掉了排序规则的主验收步骤。`
- `产物不满意：计时器列表的展示结果缺少运行中与即将结束计时器同时存在时的排序证据，无法证明 TimerList 会把最需要关注的计时器排在前面，不能满足“更不容易盯错计时器”的需求。`
- `过程不满意：在编辑 auth.py 的步骤中，模型没有先读取最新文件上下文就直接改登录逻辑，导致新增的 token 变量没有来源；正确做法应先确认现有鉴权参数，再改调用链。`
- `产物不满意：产物 auth.py 的 login() 使用未定义变量 token，运行登录流程会触发 NameError，无法满足“登录后保存会话状态”的需求。`
- `过程不满意：在终端返回 npm install 版本冲突后，模型连续 3 次重复执行相同安装命令，没有读取 package.json 排查依赖版本，导致问题没有被推进。`
- `产物不满意：prompt 要求支持分页，但产物只实现了列表查询，接口没有处理 page 和 pageSize 参数，无法满足分页浏览结果的需求。`
- `过程不满意：在第 5 步改完核心逻辑后，模型没有运行项目测试就宣称完成，漏掉了新增分支的真实执行验证。`
- `产物不满意：utils/date.py 的 parse() 未处理带 Z 的 ISO 时间串，输入 2026-06-08T10:00:00Z 会抛 ValueError，无法满足跨时区时间解析需求。`

## 默认输出格式

对用户输出总则：

- 每次回复默认保持简洁，只说当前结果、下一步、需要用户确认的事项；不要把脚本过程、长命令输出、内部判断链路铺开。
- 重点信息必须前置；有问题或需要用户确认时，用 `【】` 包裹，例如 `【需要确认】`、`【推送失败】`、`【缺 Trae Session ID】`、`【Excel 已回填】`。
- 没有问题时也要给出关键收尾状态，但不写大段总结。
- 发送提示词时，只给发送要求和提示词正文；发送要求第一行必须以 `【第 X 个主提示词，第 Y 轮】` 开头。
- 验收结束时，必须给回填摘要，包含是否完成、不满意原因、Commit ID、远程推送是否成功、Excel 是否已回填。
- git push 未成功或 Commit ID 未回填时，不能说“已完成收尾”，必须明确写 `【推送失败】`、`【Commit ID 未回填】` 或对应阻塞点。
- 如果需要用户确认才能继续，回复第一行必须出现 `【需要确认】`。

生成首轮主提示词并写入 Excel 时，提示词字段只保存提示词正文，不夹带任务类型、基线状态、去重说明、过程说明或额外总结。等到需要用户手动发送时，必须额外加发送说明头：`【第 X 个主提示词，第 Y 轮】`。

到“需要用户手动发送 Trae 提示词”的节点时，默认输出改为下面两种之一：

```text
【第 X 个主提示词，第 1 轮】【需要新建任务】请发送这条主提示词：
<提示词正文>
```

或：

```text
【第 X 个主提示词，第 Y 轮】【不要新建任务】请在当前任务下发送这条修复提示词：
<提示词正文>
```

用户发送完成后，Agent 默认要求用户立刻回复：

```text
【需要确认】发送完成后，请粘贴 Trae Session ID：<完整值>
```

后续修复提示词或验收场景：

```text
【验收完成】
是否完成：已完成
过程与产物是否满意：满意
修改范围：<单文件/模块内多文件/跨模块多文件/跨系统多模块/无需修改>
不满意原因：无
Commit ID：<40 位完整 SHA 或 无代码改动>
远程推送：成功/失败/无需推送
Excel 回填：已完成/未完成
下一步：<发送下一条/等待确认/流程暂停>
```

或：

```text
【验收未完成】
是否完成：未完成
过程与产物是否满意：不满意
修改范围：<单文件/模块内多文件/跨模块多文件/跨系统多模块/无需修改>
不满意原因：
过程不满意：...
产物不满意：...
Commit ID：<40 位完整 SHA 或 无代码改动>
远程推送：成功/失败/无需推送
Excel 回填：已完成/未完成
下一步：<发送修复提示词/进入下一条/等待确认/流程暂停>
```

最终输出里不能出现 `是否完成：暂时无法判定完成`。

## 脚本自检和 dry-run

静态检查：

```bash
python3 -m py_compile $CODEX_HOME/skills/solo-faster/scripts/*.py
```

v2 dry-run 自检：

```bash
python3 $CODEX_HOME/skills/solo-faster/scripts/solo_faster_v2_dry_run.py
```

dry-run 覆盖：

- 创建默认分布的主提示词行。
- 模拟第 1 条已完成并进入第 2 条。
- 模拟第 1 条失败后生成第 2 轮修复。
- 模拟第 3 轮失败后停止修复。
- 模拟 Bug 修复数量达到当前 workbook 的 `0-1代码生成` 总数后只跑首轮。
- 模拟超时待人工。
- 模拟断点续跑。
- 模拟真实 git commit 并校验 40 位 `Commit ID`、commit message 等于 `Trae Session ID`、Excel 已回填。
- 反例校准：短 commit、随机 commit、message 不等于 Session ID、push/commit 后漏回填、已跳过但实际执行、满意但存在不满意原因、提交后修改范围为无需修改、不满意原因含验收背景描述、不满意原因跨行长片段雷同，必须全部被 `audit` 拦截。
