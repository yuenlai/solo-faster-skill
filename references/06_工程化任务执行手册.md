# 工程化任务执行手册

## 出题目标

围绕构建流程、自动化、CI/CD、测试配置、依赖管理等工程效率问题出题，描述团队在开发流程中遇到的痛点，让模型提供可落地的工程化方案。

题目读起来像团队负责人在解决开发效率问题，而不是工具文档说明。

---

## 提示词质量标准

**必须包含：**
- 当前工程流程中的痛点（手动操作、配置混乱、发布风险等）
- 期望达到的改进效果（自动化、标准化、减少人为错误等）
- 与现有技术栈/流程的兼容性要求

**禁止包含：**
- 文件路径、配置文件名、脚本名称
- 具体工具命令（"用 GitHub Actions""写 Makefile target"）
- Markdown 格式

**字数要求：**
- 全文不超过 80 字（空白字符不计入），约束要求融入正文，不得单独分行或加标签前缀

---

## 正确示例

每次发布新版本都需要人工检查依赖版本、手动跑测试、手动打包，步骤多容易遗漏，不同人操作结果也不一样。希望把这些步骤标准化并自动触发，发布前自动完成检查和构建，减少人工干预。

---

## 反面示例（禁止出现此类写法）

❌ 请在 .github/workflows/ 目录下创建一个 CI 流程文件，包含 lint、test、build 三个 job，在 push 到 main 分支时自动触发。

❌ 修改 Makefile，添加 release target 调用 docker build 和 docker push。

---

## 常见错误

1. 直接写出工具名或配置文件路径 → 改用"自动化流程""发布检查"等业务描述
2. 需求太抽象（"改善 CI"）→ 补充具体的流程痛点
3. 涉及业务逻辑而非工程化 → 工程化只关注流程、工具链、自动化，不涉及业务
4. 超过 80 字 → 删去背景，直接说"现状问题 + 期望效果"
