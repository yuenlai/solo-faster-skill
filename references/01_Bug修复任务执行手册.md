# Bug 修复任务执行手册

## 出题目标

找出代码中存在的真实缺陷（逻辑错误、边界遗漏、运行时异常、数据异常等），以用户视角描述触发场景和异常现象，让模型去定位并修复。

题目读起来像真实用户在反馈问题，而不是工程师在指派任务。

---

## 出题前置：bug 必须真实且可复现

**先读代码找 bug，再翻译成用户语言。不得反过来：不得先想一个合理的 bug 故事再套到项目上。**

出题前必须能在代码里同时指认清楚：

- 触发代码：文件名 + 行号 + 相关函数
- 观察点：用户在 UI 或接口返回上能直接看到的异常
- 最短复现步骤：用户从干净状态开始，几步操作能触发

硬性门槛：

- 最短复现步骤 > 3 步用户操作 → 换一个 bug；确实没有更短的，则在题目"触发场景"里把每一步都写全
- 复现依赖特殊前置状态（session 过期、脏 localStorage、特定角色数据、并发等）→ 前置动作必须写进题目，不得默认用户自己能想到
- bug 横跨多个文件/模块（例如某个视图看起来正常，真实根因在 store 清理或路由守卫里）→ 出题前必须通读完整调用链，不得只凭单个文件的表象就下笔

自问三遍，任何一条答不上来就重写题目：

1. 新克隆仓库、干净环境下，按题目步骤走一遍能不能看到题目描述的现象？
2. 题目里的"当前行为"是代码真实行为，还是我脑补的 bug 剧本？
3. 被测模型只看题目、不看我的分析，能不能定位到我心里选定的那段代码？

---

## 提示词质量标准

**必须包含：**
- 用户遇到了什么问题（异常现象、错误结果、功能失效）
- 触发该问题的操作路径或场景
- 期望的正确行为（对比当前错误行为）

**禁止包含：**
- 文件路径、函数名、类名、变量名
- 数据库表名、字段名、API 路径字符串
- "修改 xxx 文件""调用 xxx 方法"等技术指令
- Markdown 格式（标题、粗体、列表符号、代码块）

**字数要求：**
- 全文不超过 80 字（空白字符不计入），约束要求融入正文，不得单独分行或加标签前缀

---

## 正确示例

用户在批量导入数据时，如果某一行格式有误，系统会直接报错中断，已经成功导入的数据也一并回滚了。期望的行为是：跳过格式错误的行并记录到错误日志，成功的数据正常保留。

---

## 反面示例（禁止出现此类写法）

❌ 请修复 ImportService.java 中 batchImport() 方法的事务处理逻辑，当 validateRow() 返回 false 时应只 rollback 当前行而不是整个事务。

❌ 修复 user_controller.go 第 142 行的空指针问题。

---

## 常见错误

1. 把文件名或方法名直接写进题目 → 删掉，改用业务描述
2. 题目过于笼统，没有现象描述 → 补充"什么操作 → 出现什么问题"
3. 超过 80 字 → 合并相似内容，保留核心现象和期望结果
4. 使用 Markdown 格式 → 改为纯文本段落
