# 代码生成任务执行手册

## 出题目标

基于现有系统的业务背景，要求从零生成一个新功能模块。以产品需求的形式描述目标，让模型自主设计并实现完整可运行的代码。

题目读起来像产品经理在提新需求，而不是工程师在分配任务。

---

## 提示词质量标准

**必须包含：**
- 新功能的业务目标（解决什么问题、满足什么需求）
- 功能的核心行为描述（输入什么、输出什么、有什么规则）
- 与现有系统的关联（接在哪个环节、数据从哪来）

**禁止包含：**
- 文件路径、类名、函数名、数据库表名
- 技术选型指令（"用 Redis 做缓存""调用 xxx API"）
- Markdown 格式（标题、粗体、列表符号、代码块）

**字数要求：**
- 全文不超过 80 字（空白字符不计入），约束要求融入正文，不得单独分行或加标签前缀

---

## 正确示例

现有系统支持用户手动提交审核申请，希望新增一个定时自动提醒功能：对于超过 3 天未处理的申请，系统每天凌晨自动给对应负责人发送站内提醒，提醒内容包含申请编号和等待天数。

---

## 反面示例（禁止出现此类写法）

❌ 请在 scheduler/reminder.go 中实现一个定时任务，使用 cron 表达式 "0 0 * * *"，查询 applications 表中 status='pending' 且 created_at < now()-3天 的记录，调用 NotifyService.send() 发送通知。

❌ 实现一个 RESTful 接口 POST /api/v1/reminders，入参包含 userId 和 message 字段。

---

## 常见错误

1. 把实现细节当需求描述 → 只说"要做什么"，不说"怎么做"
2. 功能边界不清晰 → 补充触发条件、处理规则
3. 缺少与现有系统的联系 → 说明新功能接在哪个流程里
4. 超过 80 字 → 合并重复表达，保留核心业务逻辑
